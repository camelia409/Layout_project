============================================================
1. PROJECT OVERVIEW
============================================================
This project generates Tamil Nadu style residential floor plans from
structured rules, synthetic training data, and trained ML models.

The architecture is hybrid:
- Rule-driven where geometry, circulation, doors, windows, setbacks,
  materials, and code checks must be deterministic and explainable.
- Model-driven where room sizing and plan validity scoring benefit from
  learned patterns from the synthetic dataset.

The target pipeline is:
CSV seed rules -> SQLite knowledge DB -> synthetic parquet dataset ->
trained models -> engine -> renderer -> Streamlit app

4. CORE RULE MAPPING
============================================================
The project intentionally splits responsibilities between DB rules,
Python geometry, and ML models.

4.1 Rules that come from the database

tn_setbacks
- Used for front, rear, and side setbacks
- Drives net buildable width and depth
- Used in Phase 4 engine geometry

climate_data
- Used for district -> climate_zone
- Used for window orientation preference:
  window_north_score, window_south_score, window_east_score,
  window_west_score
- Used in engine window placement

movement_paths
- Encodes critical circulation logic
- Informs required room-to-room access expectations
- Used to derive door connectivity priorities

passage_dimensions
- Supplies door widths and passage clear widths
- Engine explicitly reads min_clear_width_m from DB
- This is the authoritative source for door widths

adjacency_rules
- Encodes MUST_SHARE_WALL and forbidden adjacency semantics
- Used to reason about kitchen-bedroom, toilet-kitchen,
  master_bedroom-toilet_attached relationships
- Also informs layout debugging and scoring logic

nbc_codes
- Encodes mandatory room minimums and code constraints
- In practice used as the source-of-truth category for minimum areas,
  widths, and compliance concepts

materials_db
- Provides district-aware, climate-aware material recommendations
- Used by engine to attach render-ready material metadata

baker_principles
- Provides climate and plot-area filtered cost-saving / sustainability
  guidance
- Used by engine as design recommendation context

plot_configurations
- Used earlier during synthetic data generation to derive room target
  dimensions and realistic plot-to-BHK sizing patterns

circulation_zones
- Seeded for future zoning logic and layout orchestration
- Likely more useful in engine refinement and app-level explanation

entry_sequence_rules
- Seeded for future entrance composition and entry progression
- Can later inform renderer annotations and front-entry design logic

4.2 Rules that stay in Python code

Wall geometry
- Exterior wall thickness: 230 mm
- Interior wall thickness: 115 mm
- These are deterministic geometry rules, not learned behavior

Wall network extraction
- Derived from room adjacency and room edges
- Needed so renderer draws walls as built elements

Door placement geometry
- Doors are placed on shared walls
- Wall midpoint, swing side, and opening placement are code-driven

Window placement geometry
- Exterior wall eligibility is computed in code
- Climate orientation preference is read from DB

Rotation / facing transforms
- North, South, East, West plot orientation transforms are handled in code

4.3 What the models do

room_dimensions.h5
- Predicts room widths and depths from:
  plot_w, plot_d, plot_area, net_w, net_d, net_area,
  bhk, facing_code, climate_code

constraint_scorer.pkl
- Scores whether a proposed arrangement resembles a valid plan from the
  synthetic dataset
- Consumes 74 numerical features from build_feature_vector()

shap_explainer.pkl
- Explains which input features most influenced the scorer output

6. ENGINE WORKFLOW (PHASE 4)
============================================================
The intended engine pipeline is:

1. Load models
- XGBoost scorer
- Keras dimension predictor
- SHAP explainer

2. Query database context
- setbacks
- climate zone
- window orientation scores
- material recommendations
- Baker principles

3. Compute net buildable envelope
- plot_w / plot_d
- minus setbacks
- result: net_w / net_d

4. Predict room dimensions
- Uses room_dimensions.h5
- Produces room-specific width/depth candidates

5. Place rooms
- Current implementation uses horizontal bands
- Future refinement must make this distributionally closer to valid
  training layouts

6. Build wall network
- Convert room rectangles into shared interior walls + exterior walls

7. Place doors
- Uses required circulation logic
- Door width comes from passage_dimensions DB

8. Place windows
- Uses climate orientation scores from climate_data
- Places windows on best exterior walls
- Places ventilators in wet rooms

9. Build scorer feature vector
- 74 columns in exact XGBoost order

10. Score and explain
- score_valid from XGBoost
- score_vastu / score_nbc / score_circulation / score_adjacency in Python
- SHAP top features for explainability

8. KNOWN ENGINE ISSUE RIGHT NOW
============================================================
Important current state:
- The engine's rule-based score_overall can reach 1.0
- But the ML scorer can still return near 0.0

Why:
- The scorer was trained on synthetic layouts with a specific spatial
  distribution
- Even if the engine plan is architecturally reasonable, if the room
  centroids / sizes / arrangement differ from the learned valid cluster,
  XGBoost will still classify it as invalid
- Additionally, the model includes plot_w/plot_d/net_w/net_d features. If
  the training set contains no valid examples for a plot-size band, the
  classifier will reject that band regardless of reasonable geometry.

Observed mismatch for current 12x15 2BHK Coimbatore test:
- score_valid is still approximately 0.0
- score_overall can be high
- This means deterministic rule checks and learned validity are not yet
  aligned
- Confirmed training distribution gap:
  - Valid 2BHK plot_area values present: 360, 432, 600 sqm
  - Valid 2BHK count for plot_area 150-220 sqm: 0

Key learned 2BHK valid pattern from training:
- masterbedr_cx_pct ~ 0.134
- masterbedr_cy_pct ~ 0.108
- toiletatta_cx_pct ~ 0.049
- toiletatta_cy_pct ~ 0.279
- bedroom2_cx_pct ~ 0.386
- bedroom2_cy_pct ~ 0.092
- living_cx_pct ~ 0.149
- living_cy_pct ~ 0.740
- kitchen_cx_pct ~ 0.818
- kitchen_cy_pct ~ 0.799
- verandah_cy_pct ~ 0.942
- toiletcomm_cx_pct ~ 0.763
- toiletcomm_cy_pct ~ 0.550

This indicates the next engine fix should be layout-distribution driven,
not just code-validity driven.
